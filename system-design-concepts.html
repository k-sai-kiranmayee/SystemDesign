Browser HTTP Caching

Cache-Control headers that let browsers store responses locally

Definition ▾

Browser HTTP caching allows a browser to store copies of responses (HTML, CSS, JS, images) locally and reuse them on subsequent requests. This reduces load times, saves bandwidth, and decreases server load. It is controlled via HTTP response headers.

How It Works ▾

Key headers:

  • Cache-Control: max-age=31536000 — cache for 1 year (for fingerprinted assets)
  • Cache-Control: no-cache — must revalidate with server before using cached copy
  • Cache-Control: no-store — never cache (sensitive data)
  • ETag: "abc123" — unique identifier for a resource version; browser sends If-None-Match on revalidation

A 304 Not Modified response means the cached copy is still valid — no body is transmitted.

Real-World Example ▾

Webpack adds a content hash to asset filenames (e.g., main.a3f9bc.js). The server sets Cache-Control: max-age=31536000, immutable. Browsers cache them forever. When you deploy new code, the filename hash changes, bypassing the cache for only the new file.

Browser Rendering Pipeline

How HTML, CSS, and JS become visible pixels on screen

Definition ▾

The browser rendering pipeline is the sequence of steps a browser follows to convert HTML, CSS, and JavaScript source files into a fully painted, visible web page. Understanding this pipeline helps developers eliminate render-blocking resources and achieve faster First Contentful Paint.

How It Works ▾

The pipeline has five key stages:

  1. Parse — Browser parses HTML into the DOM and CSS into the CSSOM
  2. Style — Combines DOM + CSSOM to compute styles for every element
  3. Layout (Reflow) — Calculates the exact position and size of every element
  4. Paint — Fills in pixels: colours, borders, shadows, text
  5. Composite — Layers are combined and sent to the GPU for display

JavaScript can block HTML parsing if a <script> tag has no defer or async attribute — a common cause of slow page loads.

Real-World Example ▾

When Google Fonts is loaded in the <head> without display=swap, the browser blocks rendering until the font downloads. Adding font-display: swap allows the browser to show fallback text immediately and swap in the custom font once loaded — a key Lighthouse optimisation.

Content Security Policy (CSP)

HTTP header that restricts what resources a page can load

Definition ▾

Content Security Policy (CSP) is an HTTP response header (or meta tag) that tells the browser which sources of content are trusted. It is a defence-in-depth mechanism against Cross-Site Scripting (XSS) attacks by blocking inline scripts and restricting resource loading to approved origins.

How It Works ▾

CSP is a whitelist of trusted sources per content type:

Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://cdn.example.com;
  img-src *;
  • 'self' — only same origin
  • 'none' — block all sources of this type
  • 'unsafe-inline' — allow inline scripts/styles (weakens XSS protection)
  • nonce-{random} — allow a specific inline script with a matching nonce attribute
Real-World Example ▾

GitHub sets a strict CSP that blocks all inline scripts and only allows scripts from its own domain. Even if an attacker finds an XSS injection point, they cannot load an external payload because the browser's CSP engine blocks the request. GitHub uses nonces to allow legitimate inline scripts written by its engineers.

CORS (Cross-Origin Resource Sharing)

Browser security policy that controls cross-origin HTTP requests

Definition ▾

CORS is a browser security mechanism that restricts HTTP requests made from one origin (domain + protocol + port) to a different origin. Browsers enforce the same-origin policy by default; CORS is the standardised way for servers to explicitly permit cross-origin requests.

How It Works ▾

For non-simple cross-origin requests, browsers send a preflight OPTIONS request first. The server responds with CORS headers:

  • Access-Control-Allow-Origin: https://app.example.com
  • Access-Control-Allow-Methods: GET, POST
  • Access-Control-Allow-Headers: Authorization, Content-Type
  • Access-Control-Max-Age: 86400 — cache preflight for 24 h

Credentials (cookies, auth headers) require Access-Control-Allow-Credentials: true and a specific origin (not *).

Real-World Example ▾

A React app at app.example.com calls a REST API at api.example.com. Without CORS headers on the API, the browser blocks every response. Adding Access-Control-Allow-Origin: https://app.example.com to API responses allows the browser to pass them to JavaScript.

Critical Rendering Path

The sequence of steps the browser takes to render the first visible frame

Definition ▾

The Critical Rendering Path (CRP) is the minimum set of steps a browser must complete before it can display the first visible content. Optimising the CRP means minimising the number and size of render-blocking resources.

How It Works ▾

The CRP stages: DOM construction → CSSOM construction → Render Tree → Layout → Paint. Resources that block this path are render-blocking:

  • Render-blocking CSS: All <link rel=stylesheet> in <head> block rendering
  • Render-blocking JS: <script> without defer/async blocks both parsing and rendering
  • Optimisations: Inline critical CSS, defer non-critical JS, use rel=preload for fonts
Real-World Example ▾

Google's PageSpeed Insights penalises render-blocking resources. A common fix is loading non-critical CSS asynchronously using the media="print" trick or inlining 1–2 KB of above-the-fold CSS directly in a <style> tag — eliminating the render-blocking request entirely.

DOM (Document Object Model)

The browser's live tree representation of an HTML document

Definition ▾

The Document Object Model (DOM) is a programming interface for web documents. It represents the structure of an HTML or XML document as a tree of nodes, where every element, attribute, and piece of text is a node that can be accessed and manipulated programmatically.

How It Works ▾

When a browser parses an HTML file, it builds the DOM tree in memory. Each HTML tag becomes a node: the <html> element is the root, <head> and <body> are children, and so on. JavaScript uses the DOM API (document.querySelector, element.addEventListener, element.textContent) to read and update this tree.

  • Node types: Element nodes, Text nodes, Comment nodes
  • Traversal: parentNode, childNodes, nextSibling
  • Mutation: appendChild, removeChild, setAttribute
  • Events: Click, input, load events bubble up the DOM tree
Real-World Example ▾

React builds a Virtual DOM — a lightweight copy of the real DOM. When state changes, React diffs the old and new Virtual DOM and applies only the minimum necessary real DOM mutations. This avoids expensive full-page re-renders and is why React apps feel fast even with frequent data updates.

HTTP Client (fetch / XHR)

Browser APIs that send HTTP requests to servers

Definition ▾

A browser HTTP client is a JavaScript API that allows web pages to make HTTP requests to servers without navigating away from the page. The modern standard is the fetch API; XMLHttpRequest (XHR) is the older predecessor. Both enable AJAX — Asynchronous JavaScript and XML — patterns.

How It Works ▾

The fetch API returns a Promise that resolves to a Response object:

fetch('/api/users')
  .then(res => res.json())
  .then(data => console.log(data));

Key concepts:

  • Request methods: GET, POST, PUT, DELETE, PATCH
  • Headers: Content-Type, Authorization, Accept
  • CORS: Cross-origin requests require server opt-in via CORS headers
  • Credentials: Cookies not sent cross-origin by default; use credentials: 'include'
Real-World Example ▾

Twitter's web app uses fetch to load new tweets as you scroll. When you reach the bottom of your timeline, the client sends a GET request to the tweets API with a cursor parameter. The server returns JSON; the client appends new tweet cards to the DOM — all without a full page reload.

IndexedDB

Browser-native NoSQL database for structured offline data storage

Definition ▾

IndexedDB is a browser-native, low-level NoSQL object store for storing large amounts of structured data — including files and blobs — on the client. Unlike localStorage (5–10 MB string-only storage), IndexedDB supports indexed queries, cursor iteration, transactions, and stores binary data. It is available inside service workers, enabling offline-capable web applications.

How It Works ▾

Resource Hints & Preloading

HTML link elements that tell the browser to fetch or connect to resources in advance

Definition ▾

Resource hints are <link> elements placed in the HTML <head> that communicate the developer's intent about future resource needs to the browser. They allow the browser to begin fetching, connecting, or downloading resources before they are explicitly needed — reducing perceived latency without blocking the page.

How It Works ▾

The four main hint types:

  • preload: Fetch a resource you'll need very soon on the current page at high priority. Use for: LCP images, fonts, critical JS/CSS loaded late. Example: <link rel="preload" href="hero.jpg" as="image">. Requires as attribute for correct priority and caching.
  • prefetch: Fetch a resource needed for a likely next navigation at lowest priority. Browser may ignore on slow connections. Example: <link rel="prefetch" href="/checkout.js">.
  • preconnect: Establish TCP + TLS connection to an origin in advance (no download). Saves 100–500 ms per origin. Example: <link rel="preconnect" href="https://fonts.googleapis.com">.
  • dns-prefetch: Resolve DNS for a domain in advance. Lighter than preconnect. Use as fallback for browsers that don't support preconnect.

Warning: Over-hinting competes with the browser's own prioritisation and can hurt performance. Only hint resources you're certain will be needed.

Real-World Example ▾

Google Fonts documentation recommends adding <link rel="preconnect" href="https://fonts.googleapis.com"> before the font stylesheet tag — saving the TCP+TLS handshake time. Next.js automatically adds <link rel="prefetch"> tags for all visible <Link> elements in the viewport, making page transitions feel instant. E-commerce sites preload product images that are about to enter the viewport during scroll.

Same-Origin Policy

The core browser security model that isolates documents from different origins

Definition ▾

The Same-Origin Policy (SOP) is the most fundamental security mechanism in the browser. It prevents a script loaded from one origin (scheme + hostname + port) from reading data from a different origin. Without SOP, any website you visit could silently read your emails, bank balance, or session tokens from other open tabs.

Two URLs share an origin only if all three components are identical: https://app.example.com:443 vs https://api.example.com:443 are different origins (different hostname).

How It Works ▾

What SOP blocks:

  • Cross-origin fetch/XHR responses (unless server sends CORS headers)
  • Cross-origin reading of document.cookie, localStorage, sessionStorage
  • Cross-origin access to iframe.contentDocument

What SOP allows (embedding, not reading):

  • Loading cross-origin images, scripts, stylesheets, iframes via HTML tags
  • Cross-origin form submissions (POST)

Mechanisms that extend SOP safely:

  • CORS: Server opts in to cross-origin reads via headers
  • postMessage: Safe cross-origin iframe communication with explicit origin check
  • document.domain: Legacy; allows same-parent-domain communication (deprecated)
  • SameSite cookies: Complement SOP by blocking cross-site cookie sending (CSRF protection)
Real-World Example ▾

Without SOP: you visit evil.com, its JavaScript does fetch('https://bank.com/balance'), reads your account data (you're already logged into bank.com), and sends it to the attacker's server. SOP blocks the fetch response from being readable by evil.com's script. CORS allows bank.com to explicitly permit its own mobile app domain to read the API response, while still blocking evil.com.

How It Works ▾

Client side:

  showNotification(JSON.parse(e.data));
});

Server side (response):

Content-Type: text/event-stream
Cache-Control: no-cache

data: Hello, world!


event: notification
data: {"type":"alert","msg":"Deploy complete"}

Service Workers & Offline Caching

Definition ▾

A service worker is a JavaScript file that runs in a background thread separate from the main page. It acts as a programmable network proxy, intercepting fetch requests and serving responses from a cache — enabling offline support, background sync, and push notifications.

How It Works ▾

  1. Register: navigator.serviceWorker.register('/sw.js')
  2. Install: Cache static assets (HTML, CSS, JS, images)
  3. Activate: Clean up old caches from previous versions

Web Workers

Background JavaScript threads that keep the UI responsive during heavy computation

Definition ▾

Web Workers allow JavaScript to run in background threads, completely separate from the browser's main thread (UI thread). How It Works ▾

Creating a dedicated worker:

// main.js
const worker = new Worker('worker.js');
worker.postMessage({ data: largeArray });
worker.onmessage = (e) => console.log('Result:', e.data);

// worker.js
self.onmessage = (e) => {
  const result = heavyComputation(e.data.data);
  self.postMessage(result);
};

Key rules:

  • Workers have no access to the DOM, window, or document
  • Communication via postMessage — data is cloned (structured clone algorithm)
  • Transferable objects (ArrayBuffer, MessagePort) can be transferred (zero-copy) for performance
  • SharedArrayBuffer + Atomics enable true shared memory between workers
  • Service Workers are a specialized type that can intercept network requests
Real-World Example ▾

Figma uses Web Workers to run their C++ rendering engine (compiled to WebAssembly) off the main thread — users can drag and resize shapes without any jank. WebAssembly (WASM)

Binary instruction format that runs near-native-speed code in the browser sandbox

Definition ▾

WebAssembly (WASM) is a binary instruction format and execution environment that runs in all modern browsers at near-native speed. It enables languages like C, C++, Rust, and Go to be compiled to a compact binary that the browser executes in a sandboxed, memory-safe environment alongside JavaScript — complementing JS for CPU-intensive workloads like image processing, games, scientific computing, and codecs.

How It Works ▾

Compilation pipeline:

  1. Write code in C/C++/Rust/Go
  2. Compile to .wasm binary using Emscripten (C/C++) or wasm-pack (Rust)
  3. Load in browser: WebAssembly.instantiateStreaming(fetch('module.wasm'))
  4. JavaScript calls exported WASM functions; WASM can call imported JS functions

WASI (WebAssembly System Interface) extends WASM to server-side runtimes (Wasmtime, WASMEdge) with file system, network, and system call access. Performance: Typically within 1.0–1.5× native C speed; much faster than equivalent JavaScript for numeric/memory-intensive workloads.

Real-World Example ▾

Figma built their entire rendering engine in C++ and compiled it to WebAssembly — this is why Figma's canvas handles thousands of objects at 60 fps in a browser tab. Google Earth Web uses WASM for 3D rendering. AutoCAD Web runs a decade-old C++ codebase via WASM. TensorFlow.js can use a WASM backend for faster ML inference. Anycast Networking

One IP address advertised from multiple locations; traffic routes to the nearest

Definition ▾

Anycast is a network addressing methodology where the same IP address is assigned to multiple servers in different geographic locations. Routing protocols (BGP) automatically direct each request to the topologically nearest server, minimising latency and providing resilience.

How It Works ▾

With anycast, all PoPs advertise the same IP prefix (e.g., 104.16.0.0/12) via BGP. Each user's ISP routes packets toward the BGP peer advertising the shortest AS path — typically the geographically nearest PoP:

  • A Tokyo user's packets → Tokyo PoP
  • A London user's packets → London PoP
  • Same destination IP; different physical servers

Anycast also provides DDoS resilience: attack traffic is distributed across all PoPs rather than concentrated at one origin.

Real-World Example ▾

Cloudflare's 1.1.1.1 resolver uses anycast. All users worldwide query 1.1.1.1, but packets route to the nearest Cloudflare PoP. A user in Sydney gets sub-5ms DNS resolution instead of crossing the Pacific. The 13 DNS root server clusters are similarly anycast — thousands of physical servers share 13 IP addresses.

REST API Design Principles

Building predictable, cacheable, and self-documenting HTTP APIs using REST constraints

Definition ▾

REST (Representational State Transfer) is an architectural style for distributed hypermedia systems, defined by Roy Fielding in 2000. A RESTful API uses standard HTTP semantics — methods (GET, POST, PUT, PATCH, DELETE), status codes, and URLs as resource identifiers — to create a uniform, cacheable, stateless interface. Well-designed REST APIs are intuitive, version-stable, and enable independent client and server evolution.

How It Works ▾

Resource-oriented design: Model your API as resources (nouns), not operations (verbs).

  • GET /users → list users | GET /users/{id} → get one user
  • POST /users → create user (returns 201 Created + Location header)
  • PUT /users/{id} → replace user (idempotent)
  • PATCH /users/{id} → partial update
  • DELETE /users/{id} → delete (returns 204 No Content)

Status codes carry semantics:

  • 2xx: Success — 200 OK, 201 Created, 202 Accepted, 204 No Content
  • 4xx: Client error — 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 429 Too Many Requests
  • 5xx: Server error — 500 Internal Error, 502 Bad Gateway, 503 Unavailable

Best practices: Use plural nouns (/orders, not /order). Filter, sort, paginate via query params (?status=active&sort=created_at&cursor=abc). Version via URL prefix (/v1/) or Accept: application/vnd.api+json;version=2. Document with OpenAPI/Swagger. Return consistent error objects: {"error": "code", "message": "human text"}.

Real-World Example ▾

Stripe's REST API is widely cited as best-in-class: consistent resource names, predictable error objects with code + message + param fields, idempotency keys on mutation endpoints, and pagination via cursor-based links. The GitHub REST API uses Link headers for pagination. Twilio's API design patterns (sub-resources, phone number as resource identifier) set industry standards for communications APIs.

BGP Routing

The inter-domain routing protocol that connects internet service providers

Definition ▾

Border Gateway Protocol (BGP) is the routing protocol that exchanges routing information between autonomous systems (ASes) — independently managed networks like ISPs, cloud providers, and large enterprises. BGP is what makes the global internet function as a single interconnected network.

How It Works ▾

Each AS has an Autonomous System Number (ASN). BGP peers exchange reachability: 'I can reach 104.16.0.0/12 via AS13335'. BGP selects the best path based on:

  • AS Path — prefer shorter paths (fewer AS hops)
  • Local Preference — operator-configured preference for outbound traffic
  • MED — hint for inbound traffic when multiple connections exist to the same AS

BGP hijacking occurs when a malicious AS announces routes it doesn't own. RPKI (Resource Public Key Infrastructure) signs route announcements to prevent this.

Real-World Example ▾

In 2010, Pakistan Telecom accidentally announced 0.0.0.0/0 via BGP — redirecting a fraction of global internet traffic through their network for ~2 hours. This famous incident illustrates BGP's power and the need for route filtering. Cloudflare uses RPKI to validate all route announcements it accepts.

CDN (Content Delivery Network)

Globally distributed servers that cache and serve content close to users

Definition ▾

A Content Delivery Network (CDN) is a geographically distributed network of edge servers that cache copies of static content (images, CSS, JS, video) close to end users. By serving content from the nearest edge node rather than a central origin, CDNs reduce latency and origin load.

How It Works ▾

CDN workflow:

  1. User requests https://cdn.example.com/logo.png
  2. DNS resolves the CDN domain to the nearest edge PoP via anycast
  3. Edge server checks its cache. Cache hit: serves directly. Cache miss: fetches from origin, caches it, then serves
  4. Subsequent requests serve from cache until TTL expires

CDNs also provide: DDoS mitigation, WAF, TLS termination, Brotli/gzip compression, HTTP/3 support.

DNS (Domain Name System)

Translates human-readable domain names into IP addresses

Definition ▾

The Domain Name System (DNS) is the internet's distributed directory that maps human-readable domain names (e.g., www.github.com) to machine-readable IP addresses (e.g., 140.82.121.4). Without DNS, every user would need to memorise IP addresses.

How It Works ▾

DNS resolution hierarchy:

  1. Browser cache — checked first (TTL-controlled)
  2. OS cache / hosts file — local override
  3. Recursive resolver — your ISP's or a public resolver (Cloudflare 1.1.1.1)
  4. Root nameservers — 13 root server clusters worldwide (anycast)
  5. TLD nameservers — handle .com, .org, etc.
  6. Authoritative nameserver — owned by the domain, returns the final IP

Record types: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail), TXT (SPF, DKIM).

Real-World Example ▾

When you type github.com in Chrome, the browser checks its DNS cache. On a cache miss, it queries Cloudflare's resolver (1.1.1.1). Cloudflare queries root → .com TLD server → GitHub's authoritative nameserver (AWS Route 53). The A record returns GitHub's IP. The lookup takes 10–50 ms; the response is cached by the TTL (e.g., 60 seconds).

Forward Proxy

A proxy that represents clients to external servers — hiding identity, filtering traffic, and caching

Definition ▾

A forward proxy is a server that sits between clients and external servers on the internet. Clients configure their requests to go through the proxy; the proxy forwards them to the destination and returns the response. Unlike a reverse proxy (which represents servers), a forward proxy represents the client — hiding the client's IP and controlling outbound access.

How It Works ▾

Request flow:

  1. Client configures OS/browser to use proxy at proxy.corp.example.com:8080
  2. Client sends GET http://external-site.com/page HTTP/1.1 to the proxy
  3. Proxy forwards request to external-site.com using the proxy's own IP
  4. Response returns to proxy, then to client

HTTPS proxying: Uses the CONNECT method to establish a tunnel. Proxy opens TCP connection to destination; client speaks TLS through the tunnel (proxy sees only encrypted data — cannot inspect).
For SSL inspection (corporate security), proxy installs a root certificate on clients and performs man-in-the-middle decryption.

Use cases:

  • Content filtering: Block social media, malware domains (corporate/school networks)
  • Caching: Cache popular resources, reduce bandwidth (ISP-level caching)
  • Anonymity: Hide client IP from external servers
  • Geo-bypass: Route traffic through a proxy in another country
Real-World Example ▾

Corporate networks use Squid or Zscaler as a forward proxy — all employee internet traffic passes through it for content filtering and logging. Charles Proxy is used by mobile developers to inspect HTTPS traffic from their apps (installs trusted cert on device). The Tor Network chains forward proxies (called onion routing) to anonymise traffic — each relay knows only the previous and next hop, not the full path.

gRPC & Protocol Buffers

High-performance contract-first RPC framework using binary serialisation over HTTP/2

Definition ▾

gRPC is an open-source Remote Procedure Call (RPC) framework developed by Google. It uses Protocol Buffers (protobuf) as its interface definition language and binary serialisation format, and transports data over HTTP/2. Compared to REST/JSON, gRPC is typically 5–10× faster in serialisation and 3–7× smaller in payload size.

How It Works ▾

Step 1 — Define a contract in a .proto file:

service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
  rpc StreamUsers (Empty) returns (stream UserResponse);
}

Step 2 — Generate stubs: The protobuf compiler generates typed client and server code in any supported language (Go, Java, Python, Node.js, etc.).

Communication patterns:

  • Unary RPC: Single request, single response (like HTTP REST)
  • Server streaming: One request, stream of responses
  • Client streaming: Stream of requests, one response
  • Bidirectional streaming: Both sides stream simultaneously

HTTP/2 benefits: Multiplexed streams (many RPCs on one connection), header compression (HPACK), binary framing. Deadline propagation: timeouts pass through the entire call chain automatically.

Real-World Example ▾

Google uses gRPC for internal microservice communication across all its products. Netflix uses gRPC for service-to-service calls where low latency and strong typing matter. The Kubernetes API server speaks protobuf internally (REST/JSON is a secondary interface). Square migrated from Thrift to gRPC and gained significant serialisation performance. gRPC-Web (with Envoy proxy) enables browser clients to call gRPC services directly.

HTTP Caching Headers

Cache-Control, ETag, and Vary — the directives that drive browser, CDN, and proxy caching

Definition ▾

HTTP caching headers are response (and sometimes request) headers that control how long a resource can be stored and reused by browsers, CDNs, and intermediary caches. Properly configured caching is one of the highest-impact performance optimisations available — it can eliminate most network requests entirely and reduce CDN origin traffic by 90%+.

How It Works ▾

Key headers:

  • Cache-Control: max-age=3600 — cache for 3600 seconds from the response time
  • Cache-Control: s-maxage=86400 — override max-age for shared caches (CDNs) only
  • Cache-Control: no-store — never cache (sensitive data: banking, health records)
  • Cache-Control: no-cache — cache allowed but must revalidate with server before use
  • Cache-Control: immutable — tell browser this URL's content will never change (safe with content-hashed URLs)
  • Cache-Control: stale-while-revalidate=60 — serve stale immediately, refresh in background
  • ETag: "v1-abc123" — server sends opaque identifier; browser sends If-None-Match: "v1-abc123"; server returns 304 Not Modified if unchanged
  • Vary: Accept-Encoding — maintain separate cached versions per encoding (gzip vs Brotli)

CDN caching: CDN respects s-maxage and Cache-Control: public. Cache invalidation via cache keys (URL + Vary headers) or explicit purge APIs.

Real-World Example ▾

GitHub Pages serves JavaScript and CSS bundles with Cache-Control: max-age=31536000, immutable — because filenames include content hashes (e.g., main.a3f9bc.js), the content truly never changes for that URL. The HTML index page is served with Cache-Control: no-cache — always revalidate to pick up the new bundle filename. Cloudflare's cache hit ratio for a typical static site reaches 95%+ with correct headers.

HTTP Versions (1.1 / 2 / 3)

Evolution of the HTTP protocol for faster, more efficient web communication

Definition ▾

HTTP (HyperText Transfer Protocol) is the application-layer protocol used to transfer web content. It has evolved through three major versions — HTTP/1.1, HTTP/2, and HTTP/3 — each iteration addressing performance bottlenecks of its predecessor.

How It Works ▾
  • HTTP/1.1 (1997): Text-based, persistent connections, but only one request per connection at a time (head-of-line blocking). Workarounds: domain sharding, resource bundling.
  • HTTP/2 (2015): Binary framing, multiplexing (multiple streams over one TCP connection), header compression (HPACK), server push. Eliminates HTTP/1.1 head-of-line blocking.
  • HTTP/3 (2022): Replaces TCP with QUIC (UDP-based). Eliminates TCP-level head-of-line blocking. Faster handshake (0-RTT reconnects). Better on lossy networks.
Real-World Example ▾

Google's web properties use HTTP/3. On a mobile network with 2% packet loss, HTTP/2 over TCP stalls all streams waiting for retransmission. HTTP/3 over QUIC handles packet loss per stream independently — a lost packet for one image doesn't delay loading the search results text.

IP Addressing & Subnetting

How IP addresses are structured and divided into subnets using CIDR notation

Definition ▾

Every device on a network has an IP (Internet Protocol) address — a numeric identifier that enables routing of packets to the correct destination. Subnetting divides a large address space into smaller, isolated subnetworks (subnets) for security, performance, and administrative control. CIDR (Classless Inter-Domain Routing) notation specifies both the network address and the prefix length (e.g., 10.0.0.0/24).

How It Works ▾

IPv4 structure: 32-bit address in four octets: 192.168.1.100. Total space: ~4.3 billion addresses (nearly exhausted). Private address ranges (RFC 1918) — not routable on public internet:

  • 10.0.0.0/8 — 16.7 million addresses (enterprises, cloud VPCs)
  • 172.16.0.0/12 — 1 million addresses
  • 192.168.0.0/16 — 65,536 addresses (home networks)

CIDR notation: The /N suffix is the network prefix length. 10.0.1.0/24 means the first 24 bits are the network (10.0.1) and the last 8 bits are hosts (0–255 = 256 addresses; 254 usable). Smaller prefix = larger subnet: /16 = 65,536 addresses; /28 = 16 addresses.

Key concepts:

  • Network address: First address of a subnet (e.g., 10.0.1.0) — identifies the subnet
  • Broadcast address: Last address (e.g., 10.0.1.255) — used for subnet-wide broadcasts
  • Gateway: Usually the first or second usable address (e.g., 10.0.1.1) — router for the subnet
  • Subnet mask: Alternative notation — /24 = 255.255.255.0
Real-World Example ▾

AWS VPCs are commonly created with a 10.0.0.0/16 CIDR block (65,536 IPs). This is divided into /24 subnets (256 IPs each) per Availability Zone — e.g., 10.0.1.0/24 for AZ-a public, 10.0.2.0/24 for AZ-a private, 10.0.3.0/24 for AZ-b public. This isolation means a security group misconfiguration in the public subnet cannot reach the private subnet's database servers.

Load Balancing

Distributes incoming traffic across multiple backend servers

Definition ▾

A load balancer distributes incoming client requests across a pool of backend servers to maximise throughput, minimise response time, avoid overloading any single server, and ensure high availability through redundancy.

How It Works ▾

Common algorithms:

  • Round Robin — sends each request to the next server in rotation
  • Least Connections — routes to the server with fewest active connections
  • IP Hash — hashes client IP for consistent server routing (session affinity)
  • Weighted Round Robin — higher-capacity servers receive proportionally more traffic

Layer 4 LBs (TCP) are faster but can't inspect HTTP. Layer 7 LBs (HTTP) can route based on URL path, headers, or cookies — enabling canary deployments.

NAT (Network Address Translation)

Maps private IP addresses to a public IP for outbound internet access

Definition ▾

Network Address Translation (NAT) modifies IP address information in packet headers as they pass through a router. It allows multiple devices with private IP addresses (e.g., 192.168.x.x) to share a single public IP address for outbound internet access.

How It Works ▾

When a private device (192.168.1.5:54321) sends a packet to the internet, the NAT device:

  1. Replaces the source IP with its public IP (e.g., 203.0.113.1)
  2. Records the mapping: 203.0.113.1:12345 ↔ 192.168.1.5:54321
  3. When the response arrives on port 12345, NAT rewrites the destination back to 192.168.1.5:54321 and forwards it

NAT conserves IPv4 addresses but breaks end-to-end connectivity for inbound connections — a key motivation for IPv6 adoption.

Real-World Example ▾

Every home Wi-Fi router performs NAT. Your laptop, phone, and smart TV all share one public IP. AWS uses NAT Gateways to let EC2 instances in private subnets make outbound API calls while remaining unreachable from the internet.

Service Mesh

Infrastructure layer for microservice communication — observability, security, and traffic control via sidecar proxies

Definition ▾

A service mesh is a dedicated infrastructure layer that handles all service-to-service communication in a microservice architecture. Instead of embedding networking logic (retries, circuit breaking, mutual TLS, tracing) in every service's code, the mesh moves this responsibility to a sidecar proxy co-located with each service. The result is uniform, policy-driven networking across hundreds of services without any application code changes.

How It Works ▾

Architecture:

  • Data plane: Sidecar proxies (Envoy in Istio/Linkerd) injected alongside each service pod. All inbound and outbound traffic routes through the sidecar.
  • Control plane: Central management component (Istiod) configures all sidecars with routing rules, security policies, and observability settings.

  • Mutual TLS (mTLS): Automatically encrypts all service-to-service traffic and authenticates service identities
  • Traffic management: Canary releases, A/B testing, fault injection, traffic mirroring via routing rules (no code changes)
  • Observability: Distributed tracing (Jaeger/Zipkin), metrics (Prometheus), access logs — all generated by the proxy
  • Resilience: Retries, timeouts, circuit breaking, health checks at the network layer
Real-World Example ▾

Lyft created Envoy (the proxy that Istio and Linkerd use) to manage their service mesh of hundreds of microservices. Airbnb uses Istio to safely roll out canary deployments — sending 1% of traffic to a new version and watching error rates before promoting. Uber uses a service mesh to enforce that only authorized services can call the payments microservice (mTLS + RBAC policy), without any payment service code changes.

TCP/IP Protocol Suite

The foundational protocol stack that powers the internet

Definition ▾

The TCP/IP suite is the set of communication protocols that defines how data is transmitted over the internet. It is named after its two most important protocols: TCP (Transmission Control Protocol) and IP (Internet Protocol). Together they handle addressing, routing, reliable delivery, and connection management.

How It Works ▾

TCP establishes a connection via a three-way handshake: SYN → SYN-ACK → ACK. Once connected, it guarantees ordered, reliable delivery using:

  • Sequence numbers — to reassemble out-of-order packets
  • Acknowledgements — receiver confirms receipt; sender retransmits on timeout
  • Flow control — prevents sender from overwhelming receiver (sliding window)
  • Congestion control — adapts to network capacity (slow start, AIMD)

UDP is connectionless — no handshake, no retransmit. Lower overhead, suitable for real-time video/audio.

Real-World Example ▾

A file download uses TCP: every byte arrives in order and intact. A Zoom video call uses UDP: if a video frame is delayed, the next frame is more useful than waiting for the old one to retransmit. Zoom's jitter buffer hides small packet losses without TCP's retransmit delay.

TLS / SSL

Cryptographic protocol that secures data in transit

Definition ▾

Transport Layer Security (TLS), previously known as SSL, is a cryptographic protocol that provides confidentiality, integrity, and authentication for data transmitted over a network. All HTTPS connections use TLS. Modern deployments should use TLS 1.3.

How It Works ▾

TLS 1.3 handshake (simplified):

  1. Client sends ClientHello with supported cipher suites
  2. Server responds with its certificate and ServerHello
  3. Both parties derive a shared symmetric key using ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)
  4. All subsequent data is encrypted with AES-GCM

Forward secrecy: Each session uses unique ephemeral keys — past traffic stays private even if the server key is later compromised. 0-RTT resumption: Reconnect without a new full handshake.

Real-World Example ▾

Let's Encrypt issues free TLS certificates automatically via the ACME protocol. A web server runs an agent that proves domain ownership by hosting a challenge file. Let's Encrypt signs a 90-day certificate. Server operators configure auto-renewal — no manual certificate management needed.

VPN & Tunneling

Encrypted tunnels that create secure private channels over public networks

Definition ▾

A Virtual Private Network (VPN) creates an encrypted tunnel between two endpoints over an untrusted public network (the internet), allowing the endpoints to communicate as if they were on a private network. Tunneling protocols encapsulate one network protocol inside another — allowing private IP traffic to traverse the public internet.

How It Works ▾

How a VPN tunnel works:

  1. Client and VPN server exchange credentials and negotiate encryption keys (IKE/TLS handshake)
  2. Client installs a virtual network interface (tun0/wg0); all traffic destined for the VPN subnet routes through it
  3. The VPN client encapsulates packets in the tunneling protocol (WireGuard/UDP, OpenVPN/UDP or TCP) and encrypts them
  4. The encrypted packet is sent to the VPN server over the public internet
  5. The VPN server decrypts and forwards the original packet to the destination

VPN types:

  • Remote access VPN: Individual user connects to corporate network (Cisco AnyConnect, WireGuard)
  • Site-to-site VPN: Connects two entire networks (office to data center; AWS VPC to on-premise)
  • Split tunneling: Only routes corporate traffic through VPN; internet traffic goes direct (faster, less load)

Protocol comparison: WireGuard (modern, minimal codebase, fastest), OpenVPN (flexible, widely supported), IPSec (enterprise standard, native OS support).

Real-World Example ▾

AWS Site-to-Site VPN connects a company's on-premise data center to their AWS VPC over IPSec tunnels — enabling on-premise servers to reach EC2 instances via private IPs. Cloudflare WARP uses WireGuard to route user traffic through Cloudflare's edge network, improving both security and performance. Corporate remote access VPNs were critical during COVID-19 remote work — Cisco AnyConnect and GlobalProtect handled massive traffic increases.

WebSocket Protocol (Network Level)

How HTTP upgrades to a persistent full-duplex TCP stream for real-time communication

Definition ▾

The WebSocket protocol (RFC 6455) upgrades a standard HTTP/1.1 connection into a persistent, bidirectional, full-duplex TCP stream. After the upgrade handshake, both client and server can send framed messages at any time with minimal overhead — dramatically more efficient than polling for real-time applications. This page covers the protocol-level mechanics; see WebSockets (Frontend) for the browser API.

How It Works ▾

The HTTP Upgrade handshake:

// Client → Server (HTTP/1.1 request)
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

// Server → Client (HTTP 101 Switching Protocols)
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

After the handshake, the TCP connection switches to WebSocket frame format:

  • Frame header: 2–10 bytes (FIN bit, opcode, mask bit, payload length)
  • Opcodes: 0x1 = text, 0x2 = binary, 0x8 = close, 0x9 = ping, 0xA = pong
  • Masking: Client frames are always masked with a 4-byte random key (prevents proxy cache poisoning)
  • Infrastructure: Load balancers must support WebSocket (Nginx: proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade); sticky sessions recommended (or use a distributed pub/sub like Redis)
Real-World Example ▾

Figma's collaborative canvas uses WebSockets to synchronise cursor positions and edits among all collaborators in real time. When one user moves a shape, their client sends a WebSocket frame to the server; the server fans it out to all connected clients in the same document. Trading platforms (Bloomberg, Robinhood) stream live price ticks over WebSockets — prices update multiple times per second without any polling.

Circuit Breaker Pattern

Stops cascading failures by short-circuiting calls to a failing dependency

Definition ▾

The Circuit Breaker pattern prevents a service from repeatedly calling a dependency that is known to be failing, allowing the failing dependency time to recover. It protects against cascading failures where a slow external service causes thread pool exhaustion in the calling service.

How It Works ▾

Three states:

  • Closed (normal): Requests flow through. If the failure rate exceeds a threshold (e.g., 50% in a 10-second window), the circuit opens
  • Open (failing): Requests are immediately rejected (fail fast) with a fallback response. No calls made to the dependency. After a timeout, transitions to Half-Open.
  • Half-Open (testing recovery): A limited number of probe requests are allowed. If they succeed, circuit closes. If they fail, circuit reopens.
Real-World Example ▾

Netflix's Hystrix wraps every inter-service call. When the Recommendations Service is slow, Hystrix detects timeouts above 50% and opens the circuit. Instead of every request waiting 5 seconds, Hystrix instantly returns a cached recommendation list (fallback). The Recommendations Service recovers without being bombarded by retries.

CQRS (Command Query Responsibility Segregation)

Separating write operations (commands) from read operations (queries) into independent models

Definition ▾

CQRS (Command Query Responsibility Segregation) is an architectural pattern that separates the write model (handling commands that change state) from the read model (handling queries that return data). Each side can be independently scaled, stored, and optimised — the write side for consistency, the read side for query performance.

How It Works ▾

Write side (Command model):

  • Receives commands: PlaceOrder, UpdateShipment, CancelSubscription
  • Validates business rules, applies changes to the write store (often relational, fully normalised)
  • Emits domain events: OrderPlaced, ShipmentUpdated

Read side (Query model):

  • Subscribes to domain events and projects them into denormalised, query-optimised views
  • Can use any store: Elasticsearch (search), Redis (session data), read-optimised SQL views, or a reporting database
  • Multiple projections for different use cases: user dashboard view, admin report view, search index

Eventual consistency: There is a propagation delay between a command being applied and the read model reflecting it. Clients should not expect to read their own writes immediately (use version numbers or optimistic UI).

Real-World Example ▾

LinkedIn's Feed system uses CQRS: write-side receives "User A liked Post B" events and stores them in Kafka; multiple read-side projectors consume these events and build personalised feed caches (Redis) and notification stores (Cassandra) for each user. Without CQRS, a single query joining all likes, comments, follows, and content for a feed would be impossibly expensive.

Event Sourcing

Storing state as an immutable sequence of events rather than current snapshots

Definition ▾

Event Sourcing is an architectural pattern where instead of storing only the current state of an entity, you store the full, ordered history of state-changing events that led to the current state. The current state is derived at any time by replaying the event sequence. This gives you a complete audit trail, enables time-travel debugging, and provides a natural backbone for event-driven systems.

How It Works ▾

Traditional storage: UPDATE accounts SET balance = 150 WHERE id = 1 — history lost.

Event sourcing storage:

{ type: "AccountOpened",  amount: 100, at: "2024-01-01" }
{ type: "MoneyDeposited", amount: 50,  at: "2024-01-05" }
→ Current balance: replay events → 100 + 50 = 150

Key properties:

  • Immutability: Events are facts — never updated or deleted
  • Event store: Append-only log (EventStore DB, Kafka, DynamoDB)
  • Snapshots: Periodically snapshot derived state to avoid replaying the entire log from the beginning (e.g., every 100 events)
  • Temporal queries: Replay events up to timestamp T to see what state looked like at any past moment
  • Pairs with CQRS: Events published from write side consumed by read-side projectors
Real-World Example ▾

Monzo Bank uses event sourcing for all financial transactions — every debit, credit, refund, and fee is an immutable event. Financial regulations require complete, tamper-proof audit trails, which event sourcing provides inherently. Git version control is conceptually event sourcing: each commit is an immutable event; the working tree is the current state derived by applying all commits. Kafka itself is an event store — systems that treat Kafka topics as the source of truth are practising event sourcing.

Idempotency & Retry Logic

Designing operations that are safe to retry without causing duplicate side effects

Definition ▾

An operation is idempotent if performing it multiple times produces the same result as performing it once. In distributed systems where network failures, timeouts, and retries are inevitable, idempotent operations ensure that a retry after a partial failure never causes duplicate side effects — like charging a customer twice, creating duplicate records, or sending duplicate emails.

How It Works ▾

HTTP methods:

  • Idempotent: GET, PUT, DELETE, HEAD (same result regardless of how many times called)
  • Not idempotent by default: POST (each call may create a new resource)

Idempotency keys for non-idempotent operations:

  1. Client generates a unique key (UUID) per operation attempt
  2. Client sends key in request header: Idempotency-Key: abc-123
  3. Server stores (key → result) in Redis/DB with a TTL
  4. On retry with same key, server returns stored result without re-executing

Retry strategies:

  • Exponential backoff: Retry after 1 s, 2 s, 4 s, 8 s... to avoid overwhelming a recovering service
  • Jitter: Add randomness to backoff delays to prevent all clients retrying simultaneously (thundering herd)
  • Retry budget: Maximum number of retries (typically 3–5); fail fast after budget exhausted
  • Idempotency in message queues: SQS delivers at-least-once — consumers must handle duplicates (check processed ID before acting)
Real-World Example ▾

Stripe requires an Idempotency-Key header on all payment creation requests. If your server creates a charge but the network drops before the response returns, you can safely retry with the same key — Stripe returns the original charge result without double-charging the customer. AWS SQS standard queues guarantee at-least-once delivery, so DoorDash's order processing service deduplicates incoming messages using a processed-order-IDs set in Redis before any action is taken.

Long Polling, SSE & WebSockets Compared

Three server-push patterns — trade-offs of simplicity, directionality, and infrastructure compatibility

Definition ▾

When a web application needs the server to push updates to the client, there are three primary patterns: Long Polling (a clever use of regular HTTP), Server-Sent Events (a persistent HTTP stream), and WebSockets (a full-duplex TCP connection). Choosing the right pattern depends on whether communication is one-directional or bidirectional, and what infrastructure constraints exist.

How It Works ▾

Long Polling:

  • Client sends a regular HTTP request and the server holds it open until new data is available (or a timeout occurs)
  • When data arrives, server responds; client immediately sends another request
  • Works over any HTTP infrastructure. No special proxy support needed.
  • High overhead: a new HTTP connection per message (TCP handshake + headers)
  • Use when: maximum compatibility is required; real-time is a secondary concern

Server-Sent Events (SSE):

  • Client opens one persistent HTTP connection; server sends text/event-stream messages
  • Unidirectional: server → client only. Client cannot send messages over the same connection.
  • Auto-reconnects on disconnect (browser handles this)
  • Works over HTTP/1.1 and HTTP/2. HTTP/2 allows many SSE streams on one connection.
  • Use when: server needs to push updates but client doesn't need to send messages (live scores, AI streaming, notifications)

WebSockets:

  • Upgrades HTTP to a full-duplex TCP stream; both sides can send frames at any time
  • Very low per-message overhead (2–10 byte frame header vs hundreds of bytes HTTP headers)
  • Requires WebSocket-aware load balancers and proxies
  • Use when: truly bidirectional real-time communication is needed (chat, collaborative editing, multiplayer games)
Real-World Example ▾

GitHub uses long polling for issue/PR notification updates — simplicity over real-time precision. OpenAI ChatGPT uses SSE to stream token responses as they're generated — each token is a separate event push. Slack and Figma use WebSockets — Slack needs the client to send messages to the server, and Figma needs bidirectional sync of cursor positions and edits among many collaborators simultaneously.

Message Queues

Async communication buffers that decouple producers from consumers

Definition ▾

A message queue is a form of asynchronous service-to-service communication. Producers send messages to a queue; consumers read and process them independently. The queue acts as a buffer, decoupling the producer's request rate from the consumer's processing rate.

How It Works ▾

Two main models:

  • Queue (point-to-point): One producer, one consumer. Each message is processed by exactly one consumer. AWS SQS, RabbitMQ queues — used for task distribution.
  • Pub/Sub: One producer publishes to a topic; many consumers subscribe. Each consumer gets a copy. Kafka topics, AWS SNS — used for event broadcasting.

Kafka is a distributed commit log — messages are retained for a configurable period, allowing consumers to replay history. RabbitMQ is optimised for low-latency task queues with acknowledgements.

Real-World Example ▾

When you upload a video to YouTube, the HTTP response returns immediately while background workers process the video. The upload service pushes a video.uploaded event to a Kafka topic. Multiple consumers subscribe: one transcodes to different resolutions, one generates thumbnails, one sends the notification email — each at its own pace.

Rate Limiting (App Level)

Controls how many requests a client can make in a time window

Definition ▾

Rate limiting controls the rate at which clients can make requests to a server or API. It prevents individual clients from overwhelming a service, protects against abuse and DoS attacks, and ensures fair resource allocation among all users.

How It Works ▾

Common algorithms:

  • Token Bucket: Each client has a bucket of tokens. Each request consumes a token; tokens refill at a fixed rate. Allows bursts up to bucket capacity.
  • Leaky Bucket: Requests enter a queue and are processed at a fixed rate. Smooths traffic but no bursts.
  • Fixed Window: Count requests in fixed time windows (e.g., 100 req/minute). Simple but allows edge-of-window bursts.
  • Sliding Window Log: Record timestamp of every request; count within the window. Most accurate but memory-intensive.

Implementation: Store counters in Redis. Return 429 Too Many Requests with Retry-After header.

Real-World Example ▾

Twitter's API limits the search endpoint to 180 requests per 15-minute window per token, using a sliding window in Redis. Each request stores a timestamp; the count of timestamps within the last 15 minutes determines the remaining quota. Apps exceeding the limit receive a 429 with a rate limit reset timestamp in the response header.

REST vs GraphQL

Two paradigms for designing HTTP-based APIs

Definition ▾

REST (Representational State Transfer) and GraphQL are two different paradigms for designing APIs. REST organises an API around resources (URLs); GraphQL organises an API around a typed schema — clients query exactly the fields they need in a single request.

How It Works ▾

REST: Resources = URLs. Operations = HTTP verbs. GET /users/42 returns a user object. Multiple resources require multiple round trips.

GraphQL: One endpoint (POST /graphql). Client sends a query specifying exact fields. Server returns only those fields.

  • Over-fetching: REST returns unused fields; GraphQL returns only requested fields
  • Under-fetching: REST needs N+1 requests for related data; GraphQL fetches in one trip
  • Caching: REST GET requests are naturally cacheable; GraphQL POST requests bypass HTTP cache
Real-World Example ▾

GitHub migrated from REST API v3 to GraphQL v4 because mobile clients were making 10+ REST calls to render one screen. With GraphQL, one query fetches the repository name, last 5 issues, and contributor count in a single round trip. The payload is ~60% smaller because unused fields are omitted.

Saga Pattern (Distributed Transactions)

Managing distributed transactions across microservices using compensating transactions

Definition ▾

The Saga pattern manages long-running business transactions that span multiple microservices — where no single ACID transaction can encompass the full operation. A saga breaks the transaction into a series of local transactions, each publishing an event or message that triggers the next step. If any step fails, the saga executes compensating transactions to undo the completed steps.

How It Works ▾

Two implementation styles:

1. Choreography-based saga: No central coordinator. Each service listens for events and decides what to do next.

OrderService → publishes OrderCreated
↓
PaymentService → listens, processes payment → publishes PaymentProcessed
↓
InventoryService → listens, reserves item → publishes ItemReserved
↓
ShippingService → listens, creates shipment

2. Orchestration-based saga: Central saga orchestrator sends commands to services.

OrderSaga → commands PaymentService → commands InventoryService → commands ShippingService

Compensating transactions (on failure):

  • If shipping fails → cancel inventory reservation → refund payment → mark order failed

Key properties: No distributed 2PC lock. Eventual consistency (not ACID). Each service has only a local ACID transaction. Must handle idempotency (messages may be redelivered).

Real-World Example ▾

Amazon's order fulfilment is a classic saga: (1) reserve inventory → (2) charge payment → (3) notify warehouse → (4) create shipment. If step 2 (payment) fails, a compensating transaction releases the reserved inventory. Uber's trip booking saga: match driver → charge rider → if driver cancels → trigger compensating refund. Failures at any step must gracefully roll back without leaving the system in an inconsistent state.

ACID vs BASE

Two consistency models — strong guarantees vs eventual consistency

Definition ▾

ACID and BASE are two competing consistency models for distributed databases. ACID provides strong guarantees — every transaction either fully completes or fully fails, and data is always consistent. BASE trades strict consistency for availability and performance at scale.

How It Works ▾

ACID:

  • Atomicity: All operations in a transaction succeed or all fail
  • Consistency: Every transaction brings the DB from one valid state to another
  • Isolation: Concurrent transactions don't interfere with each other
  • Durability: Committed data persists even after a crash

BASE:

  • Basically Available: System remains available during partial failures
  • Soft State: Data may change over time as replicas converge
  • Eventually Consistent: All replicas converge to the same value given no new updates
Real-World Example ▾

A bank transfer uses ACID: both balance - 100 and balance + 100 succeed or neither does. Amazon's shopping cart uses BASE (DynamoDB): adding an item might not immediately appear if you load the cart from a different region, but within seconds all replicas converge to the latest state.

CAP Theorem

A distributed system can guarantee only two of: Consistency, Availability, Partition Tolerance

Definition ▾

The CAP Theorem (proved by Eric Brewer in 2000) states that a distributed data store can only guarantee two of the following three properties simultaneously:

  • Consistency (C): Every read receives the most recent write, or an error — no stale reads
  • Availability (A): Every request receives a non-error response (not necessarily the latest data)
  • Partition Tolerance (P): The system continues operating even when network partitions occur (messages between nodes are lost or delayed)

Since network partitions are an unavoidable reality in distributed systems, the real choice is between CP (choose consistency over availability during partitions) and AP (choose availability over consistency during partitions).

How It Works ▾

CP systems: During a network partition, some nodes refuse requests to preserve consistency. A node that can't confirm it has the latest data returns an error rather than a potentially stale response.

AP systems: During a partition, all nodes continue to serve reads and accept writes. After the partition heals, nodes reconcile diverged state (via vector clocks, last-write-wins, or CRDTs).

PACELC theorem (extended nuance): Even without a partition, latency vs consistency trade-offs exist. AP systems (Cassandra) can be tuned: CONSISTENCY ONE (fast, may be stale) vs CONSISTENCY QUORUM (slower, consistent). Important caveat: CAP is a theoretical simplification. Real systems exist on a spectrum and can be tuned per operation.

  • CA (no partition tolerance): traditional single-node RDBMS — trivially consistent and available, but not distributed
  • CP: HBase, ZooKeeper, MongoDB (with majority writes), Spanner
  • AP: Cassandra, DynamoDB (default), CouchDB, Riak
Real-World Example ▾

Amazon DynamoDB defaults to AP: during a partition, it serves potentially stale reads to guarantee availability. This is appropriate for a shopping cart — showing a slightly stale count is better than showing an error. Amazon ZooKeeper (used for distributed coordination) is CP — if a partition means ZooKeeper can't reach a quorum, it refuses to elect a leader rather than potentially elect two leaders (split-brain).

Definition ▾

How It Works ▾

Row store vs column store:

Row store (PostgreSQL):    [id=1, name="Alice", age=30], [id=2, name="Bob", age=25]
Column store (Redshift):   [id:  1,2,3...], [name: Alice,Bob,...], [age: 30,25,...]

Why columns are faster for analytics:

  • I/O reduction: A query on 3 of 100 columns reads 3% of the data a row store would
  • Compression: Column values are the same data type and often sorted → high compressibility (run-length encoding, dictionary encoding, delta encoding)
  • Vectorised execution: Operations process batches of same-type values using CPU SIMD instructions

Consistent Hashing

Distributing keys across nodes on a ring so adding/removing nodes remaps only a fraction of keys

Definition ▾

Consistent hashing is a distributed hashing technique that minimises key remapping when the number of servers in a cluster changes. Traditional modular hashing (key mod N) remaps nearly all keys when N changes. Consistent hashing places both keys and servers on a conceptual hash ring (0 to 2^32 - 1) — adding or removing a server remaps only the keys that were "owned" by that server's segment of the ring.

How It Works ▾

The ring concept:

  1. Hash each server to a position on a circular ring using a hash function (e.g., MD5 of server IP:port)
  2. To find which server owns a key: hash the key, walk clockwise on the ring until you find a server
  3. Adding a server: Only keys between the new server and its predecessor move to the new server
  4. Removing a server: Only the removed server's keys move to the next server clockwise

Virtual nodes (vnodes): A single physical server is mapped to multiple (e.g., 150) positions on the ring. This ensures: (a) more uniform key distribution, (b) a failed server's load is spread across many servers rather than burdening one. Cassandra uses 256 virtual tokens per node by default.

Replication: A key is replicated to the next N servers clockwise (replication factor N). In Cassandra, replication factor = 3 means data lives on 3 nodes.

Real-World Example ▾

Apache Cassandra uses a token ring (consistent hashing) to decide which nodes own which rows — each row's partition key is hashed to a token, and the node responsible for that token range stores the row. Adding a new Cassandra node only moves ~1/N of the data, not all of it. Amazon DynamoDB uses consistent hashing to distribute partition keys across its underlying storage nodes, maintaining predictable O(1) read/write performance as the table grows.

Database Replication

Copying data to replica servers for read scaling and high availability

Definition ▾

Database replication copies and maintains data across multiple database servers. Replicas keep a copy of the primary server's data, synchronised either synchronously or asynchronously. Replication enables read scaling, high availability, and geographic distribution.

How It Works ▾

Leader-Follower (Primary-Replica): All writes go to the primary. Changes are sent to replicas via a replication log (WAL in PostgreSQL). Replicas are read-only.

Synchronous: Primary waits for at least one replica to confirm before committing — zero data loss but higher write latency.

Asynchronous: Primary commits immediately; replicas lag behind — lower write latency but risk of data loss on primary failure.

Multi-Primary: Multiple nodes accept writes; conflicts must be resolved. Used in CockroachDB, Galera Cluster.

Real-World Example ▾

GitHub's PostgreSQL cluster has one primary and multiple read replicas. Web requests that don't need up-to-the-second data hit a read replica. During a primary failure, GitHub promotes a replica using Orchestrator (automatic failover) in under 30 seconds.

Database Sharding

Horizontally partitioning data across multiple database servers

Definition ▾

Database sharding is a horizontal scaling technique that partitions data across multiple independent database servers (shards). Each shard holds a subset of the total data. Together, the shards form the complete dataset. Sharding allows databases to scale beyond what a single server can handle.

How It Works ▾

  • Range-based: user_id 1–1M on shard 1, 1M–2M on shard 2. Simple but risks hot shards.

  • Geographic: Shard by region. Reduces latency; simplifies data residency compliance.

Challenges: Cross-shard joins are expensive. Distributed transactions require two-phase commit. Resharding when adding new shards.

Real-World Example ▾

Discord shards its database by guild_id. Each Discord server (guild) always lives on the same database shard. When a message is sent, the app hashes the guild_id to find the right shard. 99% of queries touch only one shard. Discord manages resharding by routing new guilds to less-loaded shards over time.

Database Transactions

Atomic units of work that either fully succeed or fully roll back

Definition ▾

A database transaction is a sequence of one or more SQL operations executed as a single logical unit of work. Transactions guarantee that a group of operations either all succeed (permanently saved) or all fail (database left unchanged) — preventing partial updates that would leave data in an inconsistent state.

How It Works ▾ 
BEGIN;
  UPDATE accounts SET balance = balance - 500 WHERE id = 1;
  UPDATE accounts SET balance = balance + 500 WHERE id = 2;
COMMIT; -- or ROLLBACK on error

Transaction isolation levels:

  • Read Committed: Only reads committed data (default in most DBs)
  • Repeatable Read: Same rows return same values within a transaction
  • Serializable: Transactions execute as if sequential — highest isolation, lowest concurrency
Real-World Example ▾

Booking.com's seat reservation uses a transaction: SELECT ... FOR UPDATE (lock the row), then UPDATE the seat to reserved. The FOR UPDATE lock prevents two users from booking the same seat simultaneously. If any step fails, ROLLBACK ensures the seat remains available.

Distributed Databases (NewSQL)

Horizontally scalable databases that preserve ACID transactions and SQL semantics

Definition ▾

NewSQL databases (also called distributed relational databases) provide the ACID transaction guarantees and SQL interface of traditional relational databases, while scaling horizontally across many nodes like NoSQL systems. They use distributed consensus protocols (Raft or Paxos) to maintain strong consistency across nodes in multiple data centers, eliminating the traditional ACID vs. scalability trade-off.

How It Works ▾

Core mechanisms:

  • Raft consensus: Data is partitioned into ranges; each range is replicated across N nodes via Raft. Writes require acknowledgment from a majority (quorum) before committing. Enables automatic failover without data loss.
  • MVCC (Multi-Version Concurrency Control): Every write creates a new version with a timestamp. Readers see a consistent snapshot without blocking writers. Readers never block writers; writers never block readers.
  • Distributed transactions: 2-phase commit (2PC) coordinates transactions across partitions, or optimistic concurrency (read set + write set with timestamp conflict detection)

Google Spanner specifics: Uses TrueTime — hardware-based atomic clocks + GPS receivers in every data center to bound time uncertainty to within ±7 ms globally. This enables externally consistent distributed transactions without distributed locking.

AWS Aurora: Decouples storage from compute — 6 copies of data across 3 AZs with shared distributed storage; replicas read from same storage (no replication lag).

Real-World Example ▾

Google Spanner powers Google F1 (the ads database) handling millions of queries per second globally with strong consistency — critical because ad billing requires correct totals even across data centers. Full-Text Search

Inverted-index engines that enable fast keyword search, relevance ranking, and faceted filtering

Definition ▾

Full-text search (FTS) engines index the content of text fields to enable fast keyword search, relevance-scored ranking, fuzzy matching, language analysis, and faceted aggregations. At their core, they use an inverted index — a data structure that maps each unique term to the list of documents containing it — enabling sub-second searches across millions or billions of documents.

How It Works ▾

Inverted index construction:

  1. Tokenisation: "Quick brown Fox" → ["quick", "brown", "fox"]
  2. Normalisation: Lowercase, remove stop words, apply stemmer (running → run) or lemmatiser
  3. Index: fox → [doc3, doc7, doc12], with term frequency and position metadata

Scoring (BM25): Documents ranked by term frequency (TF), inverse document frequency (IDF), and field length normalisation. A rare term in a short document scores higher than a common term in a long one.

Elasticsearch architecture:

  • Index divided into shards (distributed across nodes); queries scatter-gather across all shards
  • Each shard has a primary + N replica shards (fault tolerance + read scaling)
  • Writes go to primary, replicated to replicas asynchronously
  • Near-real-time: writes go to an in-memory buffer → refreshed to a searchable segment every 1 s (default)
  • Aggregations: Faceted navigation (count by category, date histogram, range filter) computed in one query pass
Real-World Example ▾

Wikipedia uses Elasticsearch to power its site-wide search across 60+ million articles in 300+ languages with sub-100 ms response times. GitHub code search indexes billions of lines of code across public repositories using a custom Elasticsearch deployment. E-commerce platforms (Shopify, Zalando) use Elasticsearch for product search with faceted navigation (brand, price range, rating) and synonym expansion.

Graph Databases

Stores data as nodes and edges — optimised for traversing relationships and network queries

Definition ▾

Graph databases store data as a network of nodes (entities) connected by edges (relationships), both with arbitrary key-value properties. They are optimised for traversal-heavy queries — finding shortest paths, degree-of-connection, cycle detection, recommendation patterns — that would require many expensive multi-join queries in a relational database.

How It Works ▾

Property graph model:

Nodes:  (Alice {name:"Alice", age:30})-[:KNOWS {since:2020}]->(Bob {name:"Bob"})
        (Bob)-[:WATCHED {rating:4.5}]->(Movie {title:"Inception"})

Cypher query language (Neo4j):

// Find all 2nd-degree connections of Alice
MATCH (alice:Person {name:"Alice"})-[:KNOWS*1..2]->(contact)
RETURN contact.name

Why graphs outperform relational for traversal:

  • Index-free adjacency: Each node stores direct physical pointers to its neighbours. Traversing a relationship is an O(1) pointer follow, not an index lookup.
  • In a relational DB, finding N-degree connections requires N expensive JOINs that grow with table size
  • In a graph DB, traversal cost depends only on the number of edges traversed, not the total number of nodes/edges

Use cases: Social networks, fraud detection, recommendation engines, knowledge graphs, network/IT infrastructure mapping, identity and access management.

Real-World Example ▾

LinkedIn uses a graph database (in-house built) to compute 1st/2nd/3rd degree connections — the "You and Alice share 12 connections" feature would require impossible multi-level JOINs in SQL at LinkedIn's scale. Facebook TAO is a custom distributed property graph store powering the entire social graph. PayPal uses graph analysis to detect fraud rings — fraudulent accounts form detectable graph patterns (clusters of new accounts sharing devices/emails/addresses).

SQL vs NoSQL Decision Framework

How to choose between relational and non-relational databases

Definition ▾

Choosing between SQL and NoSQL databases is one of the most consequential decisions in system design. The right choice depends on your data model, query patterns, consistency requirements, and scale. There is no universally superior option.

How It Works ▾

Choose SQL when:

  • Data has complex relationships requiring multi-table joins
  • Strong ACID guarantees are required (financial transactions, inventory)
  • Schema is well-defined and stable
  • Complex queries with aggregations and GROUP BY

Choose NoSQL when:

  • Data is hierarchical / document-shaped (JSON-like, no joins)
  • Write throughput exceeds what a single SQL node can handle
  • Schema evolves rapidly
  • Simple key-based lookups dominate
  • Horizontal scaling to petabyte scale is required
Real-World Example ▾

Airbnb uses PostgreSQL for booking and financial data (ACID required), MongoDB for listing content (flexible schema — a treehouse has different attributes than a city apartment), and Redis for sessions. The choice of database is made per use case, not per organisation.

Time-Series Databases

Databases optimised for storing and querying sequential data points indexed by timestamp

Definition ▾

A time-series database (TSDB) is specialised for storing and querying time-stamped data points — metrics, events, sensor readings, financial ticks, and log streams. Unlike general-purpose databases, TSDBs are designed around the access patterns of time-series data: writes are almost always sequential (append-only), queries aggregate over time ranges (SUM, AVG, MAX over the last hour), and old data is routinely downsampled or expired.

How It Works ▾

Data model (InfluxDB/Prometheus):

measurement="cpu_usage"
tags:   {host="web-01", region="us-east"}
fields: {value=72.3}
time:   2024-01-15T14:30:00Z

Storage optimisations:

  • Delta encoding: Store differences between consecutive timestamps (timestamps are monotonically increasing — deltas are tiny)
  • XOR compression (Gorilla algorithm, Facebook): Adjacent float64 values share leading bits — XOR compression achieves 1.37 bytes/point average
  • Run-length encoding: Repeated values compressed into (value, count) pairs

Key features:

  • Retention policies: Automatically delete data older than N days/months
  • Continuous aggregates: Pre-compute hourly/daily rollups as data arrives
  • Query languages: InfluxQL/Flux, PromQL (Prometheus), SQL (TimescaleDB)
Real-World Example ▾

The Prometheus + Grafana stack is the standard monitoring solution for Kubernetes — every pod, node, and service emits metrics scraped by Prometheus every 15 seconds; Grafana renders dashboards with PromQL queries. InfluxDB is used by industrial IoT companies to store sensor readings from thousands of factory machines (temperature, pressure, vibration) at sub-second granularity. NYSE stores every trade tick (billions per day) in a TSDB for real-time trading analytics.

Write-Ahead Log (WAL)

Append-only durability log that enables crash recovery, replication, and point-in-time restore

Definition ▾

A Write-Ahead Log (WAL) is a durability mechanism where every database change is first written to an append-only sequential log on durable storage before being applied to the main data files. On crash recovery, the database replays the WAL to reconstruct all committed transactions that hadn't been flushed to the main data files. WAL also enables replication (streaming WAL to standbys) and point-in-time recovery.

How It Works ▾

Write flow:

  1. Transaction modifies data in memory (buffer pool / page cache)
  2. WAL record written to disk (sequential append — fast)
  3. WAL flushed (fsync) — only then is the transaction reported as committed (durability guarantee)
  4. Modified pages flushed to data files later (background writer, checkpoint)

Crash recovery:

  • Database starts, reads WAL from last checkpoint
  • Redo pass: replay committed WAL records not yet in data files
  • Undo pass: roll back uncommitted transactions found in WAL

Replication with WAL:

  • Primary streams WAL records to standbys in real time (streaming replication)
  • Standbys apply WAL records to stay current (logical or physical replication)
  • Logical replication: WAL decoded to SQL-level events, allowing cross-version replication

Kafka as a distributed WAL: Kafka topics are immutable, append-only, replicated logs — consumers read at any offset, enabling event sourcing and CQRS projections.

Real-World Example ▾

PostgreSQL WAL enables pg_basebackup (physical backup) + WAL archiving → PITR to any timestamp. A production database can be restored to the exact state it was in 5 minutes before a bad migration. CockroachDB uses a distributed Raft log (same concept as WAL, but distributed across multiple nodes with leader election). Binlog replication in MySQL works similarly — the binary log (MySQL's WAL) streams row-level changes to read replicas.

ARP & DHCP

Protocols that resolve IP addresses to MAC addresses and dynamically assign IPs on a network

Definition ▾

ARP (Address Resolution Protocol) operates at the Layer 2/3 boundary to resolve a known IP address to its corresponding MAC address on a local network segment. Without ARP, a device cannot build the Ethernet frame needed to deliver a packet to a local neighbour. DHCP (Dynamic Host Configuration Protocol) operates at Layer 7 over UDP to automatically assign IP addresses, subnet masks, default gateways, and DNS servers to devices when they join a network.

How It Works ▾

ARP process:

  1. Device A wants to send to IP 192.168.1.5 but doesn't know its MAC address
  2. A broadcasts ARP Request: "Who has 192.168.1.5? Tell 192.168.1.2 (A's MAC)" — sent to FF:FF:FF:FF:FF:FF (broadcast MAC)
  3. Device B (192.168.1.5) replies with ARP Reply (unicast): "192.168.1.5 is at AA:BB:CC:DD:EE:FF"
  4. A caches the result in its ARP table (TTL ~20 minutes)

DHCP process (DORA):

  1. Discover: Client broadcasts DHCP Discover (no IP yet, src=0.0.0.0, dst=255.255.255.255)
  2. Offer: DHCP server responds with an IP offer: "10.0.1.42/24 is available"
  3. Request: Client broadcasts acceptance: "I'll take 10.0.1.42"
  4. Acknowledge: Server confirms the lease (with lease duration, gateway, DNS)

Security: ARP has no authentication — ARP spoofing allows a malicious device to respond to ARP requests with its own MAC, enabling man-in-the-middle attacks. Mitigated by Dynamic ARP Inspection (DAI) on managed switches.

Real-World Example ▾

Every time a device connects to a Wi-Fi network, DHCP assigns it an IP address within seconds. AWS EC2 instances receive their private IP via DHCP from the VPC's DHCP service on boot. ARP spoofing is a real attack vector on shared networks — corporate switches use DAI and 802.1X authentication to prevent it. Cloud environments bypass traditional ARP by having hypervisors handle MAC-to-IP mapping centrally (Amazon VPC uses a virtual router that responds to all ARP requests).

How HTTP Maps to OSI Layers

Tracing an HTTP request through all seven OSI layers

Definition ▾

An HTTP request to a website traverses all seven OSI layers. Understanding this mapping demystifies how a URL typed in a browser becomes bytes on a wire and how the server reconstructs the original request. This synthesis connects the conceptual OSI model to real-world protocols.

How It Works ▾

Tracing GET https://github.com/ through all layers:

  • L7 Application: Browser creates HTTP/2 request: GET / HTTP/2 with headers (Host, Accept, etc.)
  • L6 Presentation: TLS encrypts the HTTP/2 frame. gzip/Brotli compresses the request body
  • L5 Session: TLS session established (or resumed from session ticket)
  • L4 Transport: TCP segment: src port 54321, dst port 443, seq numbers
  • L3 Network: IP packet: src IP (your IP), dst IP (140.82.121.4)
  • L2 Data Link: Ethernet frame: src MAC (your NIC), dst MAC (your router)
  • L1 Physical: Frame encoded as electrical signals on Cat6 or radio waves on Wi-Fi
Real-World Example ▾

Wireshark captures all layers simultaneously. A captured packet for an HTTPS request shows: Ethernet II frame (L2) containing IPv4 packet (L3) containing TCP segment (L4) containing TLS record (L6) containing HTTP/2 frame (L7). Engineers use this to diagnose whether an issue is at the network (L3/L4) or application (L7) level.

ICMP & Network Diagnostics

Layer 3 control protocol powering ping, traceroute, and Path MTU Discovery

Definition ▾

ICMP (Internet Control Message Protocol) is a Layer 3 protocol used by network devices to send error messages and operational information about IP packet processing. Unlike TCP/UDP, ICMP carries no application data — it exists purely for network diagnostics and error reporting. The ubiquitous ping and traceroute tools are built entirely on ICMP.

How It Works ▾

ICMP Echo (ping):

  1. Source sends ICMP Type 8 (Echo Request) to destination IP
  2. Destination replies with ICMP Type 0 (Echo Reply)
  3. RTT (Round-Trip Time) is measured; 3–5 probes sent to detect packet loss

ICMP Time Exceeded (traceroute):

  1. Traceroute sends UDP/ICMP packets with TTL=1; first router decrements TTL to 0, discards packet, and sends ICMP "Time Exceeded" back → reveals first hop's IP
  2. Repeat with TTL=2, 3, 4... revealing each hop in the path until destination reached
  3. Result: full path from source to destination with per-hop RTT

Other important ICMP messages:

  • Destination Unreachable: Port closed, network unreachable, host unreachable
  • Fragmentation Needed (ICMP Type 3, Code 4): Packet too large for next link; used by Path MTU Discovery to find the maximum packet size for a path
  • Redirect: Router tells source to use a better route

ICMP blocking: Many cloud/enterprise firewalls block ICMP for "security" — meaning ping fails even when the service is up. This breaks Path MTU Discovery, causing mysterious connection hangs.

Real-World Example ▾

When AWS marks an EC2 instance as "running" but your application can't connect, ping (ICMP) may be blocked by the security group — you need to check TCP connectivity instead. traceroute is essential for diagnosing where in the network path packets are being dropped or delayed — a useful first step when a user in a specific region reports slow performance. MTR (My Traceroute) combines ping and traceroute into a continuous display, showing which hop is causing intermittent packet loss.

IPv6 & Dual-Stack Networking

The 128-bit address successor to IPv4, and how dual-stack enables gradual migration

Definition ▾

IPv6 is the sixth version of the Internet Protocol, providing 128-bit addresses (3.4 × 10^38 total) to solve the exhaustion of IPv4's 32-bit address space (~4.3 billion addresses, effectively exhausted since 2011). Dual-stack is the transition mechanism where hosts and networks run both IPv4 and IPv6 simultaneously, allowing gradual migration without a hard cutover.

How It Works ▾

IPv6 address format: Eight groups of four hexadecimal digits separated by colons: 2001:0db8:85a3:0000:0000:8a2e:0370:7334. Abbreviated by omitting leading zeros and collapsing one run of all-zero groups with ::: 2001:db8:85a3::8a2e:370:7334.

Key IPv6 improvements over IPv4:

  • No NAT required — every device can have a globally unique address
  • Simplified header format (fewer fields) → faster routing
  • No broadcast — replaced by multicast (for group communication) and anycast (for nearest-node routing)
  • SLAAC (Stateless Address Autoconfiguration): Devices can self-assign a globally routable address from a router advertisement prefix without DHCP
  • NDP (Neighbour Discovery Protocol): IPv6 replacement for ARP — uses ICMPv6 multicast instead of broadcast

Dual-stack deployment: A dual-stack host has both an A record (IPv4) and AAAA record (IPv6) in DNS. Happy Eyeballs algorithm (RFC 8305): Browser simultaneously tries IPv6 and IPv4 connections, with IPv6 preferred but a 250 ms head start. If IPv6 doesn't respond first, IPv4 is used. This prevents IPv6 from causing slowdowns when networks have broken IPv6.

Real-World Example ▾

As of 2025, Google reports ~47% of its traffic is delivered over IPv6. T-Mobile USA operates an IPv6-only network for mobile devices (with NAT64 for legacy IPv4 content access). AWS VPCs support dual-stack mode — resources get both private IPv4 and IPv6 addresses. The Internet Exchange Points in major cities peer both IPv4 and IPv6 BGP routes. GitHub, Cloudflare, and Fastly all serve traffic over both protocols.

Layer 1 — Physical

Definition ▾

The Physical Layer (Layer 1) of the OSI model is responsible for transmitting raw bits — streams of 0s and 1s — over a physical medium. It deals with electrical, optical, or radio signals, connectors, voltages, modulation schemes, and transmission rates.

How It Works ▾

  • Transmission medium: Copper (Ethernet), fibre optic (single-mode, multi-mode), wireless radio (Wi-Fi, cellular)

Layer 2 — Data Link

Node-to-node data transfer using MAC addresses and framing

Definition ▾

The Data Link Layer (Layer 2) provides node-to-node data transfer between two directly connected devices on the same network segment. It organises raw bits from Layer 1 into frames, adds physical (MAC) addresses for device identification, and handles error detection for in-transit frames.

How It Works ▾

Layer 2 responsibilities:

  • Framing: Wraps data in a frame with a header (source/dest MAC, type) and trailer (CRC checksum)
  • MAC addresses: 48-bit hardware addresses (e.g., 00:1A:2B:3C:4D:5E). Unique per device on a local network.
  • Error detection: CRC in the frame trailer. Corrupted frames are dropped.
  • Access control: CSMA/CD (Ethernet) prevents collisions on shared media

Network switches operate at Layer 2: they learn which MAC is on which port, then forward frames only to the destination port.

Real-World Example ▾

An Ethernet frame for a web request: destination MAC is the default gateway (your router), source MAC is your laptop's NIC. The switch checks its MAC address table and forwards the frame only to the router port. Once the frame crosses the router, the MAC addresses change for each hop — but the IP addresses remain the same.

Layer 3 — Network

Routes packets across networks using IP addresses

Definition ▾

The Network Layer (Layer 3) is responsible for logical addressing and routing — forwarding packets from a source host to a destination host across multiple networks. IP (Internet Protocol) is the primary Layer 3 protocol. IP addresses identify devices globally across the internet, while MAC addresses identify devices on a local network.

How It Works ▾

Layer 3 responsibilities:

  • IP addressing: IPv4 (32-bit, e.g., 192.168.1.1), IPv6 (128-bit)
  • Packet forwarding: Routers examine the destination IP and forward packets using routing tables
  • Routing protocols: OSPF (within an AS), BGP (between ASes)
  • Fragmentation: Large packets are split to fit the MTU of intermediate links
  • TTL: Hop counter decremented by each router; packet dropped at 0 (prevents infinite loops)
Real-World Example ▾

A traceroute to google.com shows 10–15 hops — each is a Layer 3 router. Each router reads the destination IP, checks its routing table, and forwards to the next hop. Each hop changes the source and destination MAC addresses (L2) but never changes the source and destination IP addresses (L3).

Layer 4 — Transport

End-to-end communication, reliability, and flow control

Definition ▾

The Transport Layer (Layer 4) provides end-to-end communication services between applications on different hosts. It uses port numbers to multiplex multiple application connections over a single IP address, and provides either reliable (TCP) or unreliable (UDP) data delivery.

How It Works ▾

Layer 4 responsibilities:

  • Port multiplexing: Port numbers (0–65535) identify applications. HTTP = 80, HTTPS = 443, DNS = 53, SSH = 22
  • Segmentation: Large data streams are divided into segments
  • TCP: Three-way handshake, sequence numbers, acknowledgements, retransmission, flow control, congestion control
  • UDP: No handshake, no retransmission. Lower overhead; used for DNS, DHCP, VoIP, gaming

A TCP connection is identified by a 4-tuple: (source IP, source port, destination IP, destination port).

Real-World Example ▾

When Chrome opens https://github.com, it creates a TCP connection: source = (your IP, ephemeral port ~54321) → destination = (140.82.121.4, 443). GitHub can have thousands of simultaneous connections to port 443, each identified by the unique source IP + source port combination.

Layer 5 — Session

Manages the setup, maintenance, and teardown of communication sessions

Definition ▾

The Session Layer (Layer 5) manages communication sessions between applications — establishing, maintaining, and terminating connections. It handles session checkpointing, synchronisation, and dialog control. In the TCP/IP model, session management is generally handled by the application layer or TLS.

How It Works ▾

Layer 5 responsibilities:

  • Session establishment: Setting up the context for an application-level exchange
  • Session maintenance: Keeping the session alive during inactivity (keep-alives)
  • Session termination: Gracefully ending the session and freeing resources
  • Checkpointing: Allowing a session to resume after an interruption
  • Dialog control: Half-duplex vs full-duplex communication

In practice, Layer 5 is largely theoretical in TCP/IP networks — TLS handles encryption and session resumption at this conceptual layer.

Real-World Example ▾

TLS session resumption operates at Layer 5 conceptually. When your browser has previously connected to GitHub, it stores a session ticket. On reconnection, it presents the ticket and the TLS handshake is abbreviated (0-RTT or 1-RTT instead of 2-RTT), resuming the secure session without a full new handshake.

Layer 6 — Presentation

Data translation, encryption, and compression

Definition ▾

The Presentation Layer (Layer 6) translates data between the application layer format and the network transmission format. It handles encryption and decryption, data serialisation and deserialisation, and compression. TLS encryption is the most significant real-world manifestation of Layer 6.

How It Works ▾

Layer 6 responsibilities:

  • Encryption/Decryption: TLS encrypts application data before transmission and decrypts on receipt
  • Data serialisation: Converting objects to transmittable formats (JSON, XML, Protocol Buffers, MessagePack)
  • Compression: gzip and Brotli reduce payload size before transmission
  • Character encoding: UTF-8, ASCII — ensures both sides interpret text the same way
  • Data conversion: Big-endian vs little-endian byte order for cross-platform compatibility
Real-World Example ▾

When Chrome sends an HTTPS request to GitHub: TLS (Layer 6) encrypts the HTTP/2 frame (Layer 7 data). The frame also contains gzip-compressed JavaScript. On the GitHub server, TLS decrypts the frame, and the HTTP/2 server decompresses the request body. The application server sees only plain, decompressed JSON.

Layer 7 — Application

Human-facing protocols that applications use to communicate

Definition ▾

The Application Layer (Layer 7) is the topmost layer of the OSI model. It provides the interface between network services and user applications. Layer 7 protocols define the format and semantics of messages exchanged between applications — HTTP, DNS, email, and file transfer protocols all live here.

How It Works ▾

Common Layer 7 protocols:

  • HTTP/HTTPS: Web browsing and API communication. Request/response with methods (GET, POST), status codes, and headers
  • DNS: Domain name resolution queries and responses
  • SMTP/IMAP/POP3: Email transmission and retrieval
  • FTP/SFTP: File transfer
  • WebSocket: Full-duplex real-time communication
  • gRPC: Remote procedure call over HTTP/2 using Protocol Buffers

Layer 7 load balancers and WAFs can inspect this layer to make routing and filtering decisions based on HTTP paths, headers, and payloads.

Real-World Example ▾

A WAF (Web Application Firewall) operating at Layer 7 inspects every HTTP request body for SQL injection patterns like ' OR 1=1 --. A Layer 4 firewall cannot do this — it only sees IP addresses and ports. Cloudflare's WAF blocks 70 billion threats per day by analysing Layer 7 HTTP payloads invisible to network-layer devices.

OSI vs TCP/IP Model

Comparing the 7-layer OSI model with the 4-layer TCP/IP practical stack

Definition ▾

The OSI model is a conceptual 7-layer framework for understanding network communication, developed by ISO in 1984. The TCP/IP model is a simpler 4-layer model that describes how the internet actually works. Real implementations follow TCP/IP; OSI is used for teaching and troubleshooting.

How It Works ▾

Layer mapping:

  • OSI L7+L6+L5 = TCP/IP Application: HTTP, DNS, SMTP
  • OSI L4 = TCP/IP Transport: TCP, UDP
  • OSI L3 = TCP/IP Internet: IP, ICMP, ARP
  • OSI L2+L1 = TCP/IP Network Access: Ethernet, Wi-Fi

The OSI model is more granular and pedagogically useful. TCP/IP is what ships in every OS and network device. Engineers use OSI layer numbers as shorthand even when working with the TCP/IP stack.

Real-World Example ▾

When a network engineer says a DDoS attack is 'at Layer 7', they mean it is HTTP-based (e.g., HTTP flood), not a volumetric packet flood (Layer 3/4). A WAF can mitigate a Layer 7 DDoS; a volumetric DDoS requires upstream traffic scrubbing. The OSI layer number is the fastest way to communicate which part of the stack is affected.

Protocol Data Units (PDUs)

The specific name for data at each OSI layer (bits, frames, packets, segments)

Definition ▾

A Protocol Data Unit (PDU) is the name for a unit of data at each OSI layer. As data travels down the OSI stack from application to physical, each layer wraps the PDU from the layer above in its own header (and sometimes trailer) — a process called encapsulation.

How It Works ▾

PDU names by layer:

  • L7 Application: Data / Message — HTTP request, DNS query
  • L4 Transport: Segment (TCP) or Datagram (UDP) — adds ports, sequence numbers
  • L3 Network: Packet — adds source/dest IP addresses
  • L2 Data Link: Frame — adds source/dest MAC addresses + CRC
  • L1 Physical: Bits — raw electrical/optical signals

On the receiving end, each layer decapsulates (strips its header) and passes the inner PDU up to the next layer.

Real-World Example ▾

When you send a GET request: HTTP data is encapsulated in a TCP segment (adds ports 54321→443), then in an IP packet (adds IPs), then in an Ethernet frame (adds MACs). At the switch, only Layer 2 is read. At the router, Layer 2 is stripped and a new Layer 2 header is added for the next hop. At the server, all headers are stripped until the application sees only the HTTP request.

TCP 3-Way Handshake

The three-step connection establishment process that synchronises TCP sequence numbers

Definition ▾

The TCP 3-way handshake is the connection establishment protocol used by TCP (Transmission Control Protocol) before any application data can be exchanged. It synchronises the initial sequence numbers (ISNs) of both sides, confirms that both can send and receive, and establishes a reliable, ordered, full-duplex channel. Every HTTP/1.1 connection, database connection, and SSH session begins with this handshake.

How It Works ▾

The three steps:

Client                              Server
  |--- SYN (seq=1000) ------------->|   [Client → SYN: "I want to connect, my ISN is 1000"]
  |<-- SYN-ACK (seq=5000,ack=1001)-|   [Server → SYN-ACK: "OK, my ISN is 5000, got your 1000"]
  |--- ACK (ack=5001) ------------->|   [Client → ACK: "Got your 5000, connection established"]
  |===== Data flows now ============|

What it establishes:

  • Both sides exchange random initial sequence numbers (ISN) — used to order all subsequent segments
  • Both sides confirm they can send AND receive
  • Connection is in ESTABLISHED state; data transfer can begin

Performance impact: The handshake adds 1 RTT (Round Trip Time) of latency before any data can flow. For a typical internet connection (50 ms RTT), this is 50 ms wasted per new TCP connection — a key reason why HTTP/1.1 keep-alive, HTTP/2 multiplexing, and QUIC (HTTP/3, which combines transport + crypto handshake) were developed.

TCP Fast Open (TFO): A TCP extension that allows data to be sent in the SYN packet on repeat connections (using a cookie from a previous handshake), reducing latency to 0-RTT data.

Real-World Example ▾

When you visit https://google.com for the first time, the browser must complete: DNS lookup (~20 ms) + TCP handshake (1 RTT ~30 ms) + TLS 1.3 handshake (1 RTT ~30 ms) before the first byte of HTML arrives. That's ~80 ms before any content — HTTP/2 multiplexes all requests on one TCP connection to pay this cost once per origin. QUIC (HTTP/3) combines the TCP and TLS handshakes into a single 1-RTT exchange, or 0-RTT for returning connections.

TLS Handshake Deep Dive

How TLS 1.3 authenticates servers, negotiates keys, and secures all application data in 1 RTT

Definition ▾

The TLS (Transport Layer Security) handshake is the protocol that negotiates a secure channel between a client and server before any application data flows. It provides three guarantees: authentication (client verifies server's identity via its certificate), confidentiality (data encrypted with session keys), and integrity (any tampering detected via HMAC). TLS 1.3 (2018) performs the handshake in a single round trip (1-RTT), down from 2 RTTs in TLS 1.2.

How It Works ▾

TLS 1.3 handshake (1-RTT):

  1. Client Hello: Client sends supported cipher suites, key share (Diffie-Hellman public key), session ticket (if resuming)
  2. Server Hello: Server picks cipher suite, sends its DH public key share
  3. Both compute shared secret: Using Diffie-Hellman key exchange — the secret is derived from both parties' key shares. Neither party ever transmits the secret over the wire.
  4. Server sends (encrypted): Certificate (server identity), CertificateVerify (signature proving it holds the private key), Finished (authenticated)
  5. Client verifies certificate: Chain validates to a trusted root CA in the OS/browser trust store; OCSP stapling proves certificate not revoked
  6. Client sends Finished: Application data can now flow over the encrypted channel

0-RTT resumption: For returning clients with a saved session ticket, data can be sent on the very first message (0-RTT). Risk: replay attacks; limited to safe (non-mutating) requests.

Certificate chain: Server cert → Intermediate CA → Root CA. Browsers ship with ~100+ trusted root CAs. Let's Encrypt is the world's largest free CA, issuing 300+ million certificates.

Real-World Example ▾

When you visit https://stripe.com, your browser verifies Stripe's certificate (signed by a CA you trust), exchanges DH keys in the Client/Server Hello, and establishes an encrypted session — all in 1 RTT (~30 ms on a typical connection). Cloudflare terminates TLS at their edge datacenters (closest to the user), reducing the handshake RTT from ~150 ms (user→origin) to ~20 ms (user→nearest Cloudflare PoP). Let's Encrypt's ACME protocol automates certificate issuance — Certbot runs as a cron job to renew certificates before they expire every 90 days.
